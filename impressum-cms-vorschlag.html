Impressum vollstaendig aus dem CMS

Analyse des Ist-Zustands und Vorschlag, wie das Impressum ohne Neu-Deployment gepflegt werden kann

Analyse: Ist-Zustand

ContentBlock-System Vorhanden

Backend komplett: ContentBlock-Entity, ContentService, AdminContentController (/api/admin/content/{key}), ContentController (/api/content/{key}). Unterstuetzt Draft/Publish-Workflow.

ImpressumEditor Vorhanden

Formular-basierter Editor fuer strukturierte Impressum-Daten (Firma, Adresse, Kontakt, Register, Geschaeftsfuehrer, USt-IdNr.). Speichert als JSON ueber /api/admin/content/IMPRESSUM.

Pages-System Vorhanden

Zweites CMS: Page-Entity mit Slug, HTML-Inhalt, Kategorien (BLOG, SEMINAR, STATIC), Status-Workflow. Admin-UI mit HTML-Editor + Split-Vorschau existiert.

ImpressumEditor nicht geroutet Fehlt

Der ImpressumEditor existiert als Datei, ist aber in App.jsx nicht als Route eingebunden. Er ist im Admin-Panel nicht erreichbar.

Frontend liest nicht vom CMS Fehlt

Impressum.jsx ist komplett hardcoded (statisches JSX). Es gibt keinen useEffect/fetch zum oeffentlichen Endpunkt /api/content/IMPRESSUM.

Fehlende Abschnitte im Editor Unvollstaendig

Der ImpressumEditor kennt nur: Firma, Adresse, Kontakt, Register, Geschaeftsfuehrer, USt-IdNr. Es fehlen: Streitschlichtung, Urheberrecht, Verantwortlich fuer Inhalte, Haftungsausschluss.

Vorschlag

A

ContentBlock-System vervollstaendigen Empfohlen

Das bestehende ContentBlock-System wird vollstaendig verdrahtet. Der ImpressumEditor erhaelt alle fehlenden Abschnitte, wird im Admin-Panel geroutet, und das Frontend liest die veroeffentlichten Daten dynamisch vom API-Endpunkt.

    Vorteile
  • + Strukturiertes Formular — kein HTML-Wissen noetig
  • + Validierung moeglich (Pflichtfelder, E-Mail-Format)
  • + Draft/Publish-Workflow bereits implementiert
  • + 90% der Infrastruktur existiert bereits
  • + Daten als JSON — saubere Trennung von Inhalt und Layout
    Nachteile
  • - Feste Struktur — neue Abschnitte erfordern Code-Aenderung
  • - Kein Freitext/Rich-Text fuer unstrukturierte Abschnitte
B

StaticPage-System nutzen (HTML-Editor)

Das Impressum wird als STATIC-Page im bestehenden Pages-System gepflegt. Der StaticPageEditor (HTML + Split-Vorschau) erlaubt voellig freie Gestaltung. Die Route /impressum liest per Slug die veroeffentlichte Page.

    Vorteile
  • + Voellig flexible Inhalte — keine Code-Aenderung noetig
  • + Neue Abschnitte jederzeit hinzufuegbar
  • + CSS-Klassen-Referenz bereits im Editor
  • + Funktioniert auch fuer Datenschutz, AGB etc.
    Nachteile
  • - HTML-Kenntnisse erforderlich
  • - Keine Feldvalidierung
  • - Anfaellig fuer Formatierungsfehler

Umsetzungsschritte (Vorschlag A)

1

ImpressumEditor erweitern

Fehlende Abschnitte zum Editor hinzufuegen: Verantwortlicher (Name + Adresse), Streitschlichtung (Textarea), Urheberrecht (Textarea), optional Haftungsausschluss (Textarea). Jeder Abschnitt wird als eigenes Feld im JSON gespeichert.

2

Route im Admin-Panel einbinden

In App.jsx eine Route /admin/impressum hinzufuegen, die den ImpressumEditor rendert. Zusaetzlich einen Navigationseintrag im Admin-Menue ergaenzen.

3

Impressum.jsx dynamisch machen

Die hardcodierten Daten durch einen useEffect + fetch('/api/content/IMPRESSUM') ersetzen. Die JSON-Daten werden in die bestehende JSX-Struktur gemappt. Fallback: bei leerem CMS die bisherigen Platzhalterdaten anzeigen.

4

Erstbefuellung im CMS

Die aktuellen Impressum-Daten (ProuDig GmbH, Musterstrasse 123, etc.) einmalig ueber den Admin-Editor eingeben und veroeffentlichen. Danach ist die hardcodierte Version obsolet.

5

Optional: Datenschutz + AGB als StaticPages

Fuer /datenschutz und /agb das Pages-System (Option B) nutzen, da diese Seiten weniger strukturiert sind. Route /seite/:slug existiert bereits im Backend (StaticPageController).
